# PRD：A股交易决策支持助手 V1 共识规划草案

## 文档目的
把现有 deep-interview 规格扩展为可执行的共识规划草案，聚焦：团队分 lane 分工、V1 具体功能/要求、lane 间输入输出与交接、验收标准与风险。本文仅做规划，不做实现。

## 0. 背景与边界
### 已确认目标
- 面向 A 股场景，V1 首要目标是**信息聚合与摘要**，服务于人工决策支持。
- V1 只覆盖：用户持仓、观察名单、选定行业/主题，不做全市场覆盖。
- 允许输出**带分数的方向性倾向**，但必须基于证据、显式不确定性、审计记录、免责声明、人工确认。

### 硬约束
- **不做自动交易**，不接券商下单，不生成可自动执行指令。
- **V1 不做全市场覆盖**。
- **V1 不接付费数据源**。
- **V1 不做实时/高频**，以日级或盘后/盘中批次为主。
- **任何倾向输出都不能是二元或命令式买卖建议**。

### 规划假设
- 当前仓库为 greenfield，除 `AGENTS.md` 与 `.omx` 工件外暂无代码。
- 当前阶段先完成方案分工、接口、验收与风险设计，再进入执行模式。

---

## 1. Outcome / V1 总体定义
### V1 要交付什么
V1 应能围绕一个**有限股票池**形成“采集 -> 清洗 -> 归档 -> 摘要 -> 分析 -> 安全审查 -> 人工确认”的闭环，输出可追溯的研究摘要与方向性倾向参考。

### V1 不追求什么
- 不追求全网/全市场完整覆盖。
- 不追求分钟级甚至秒级监控。
- 不追求自动化执行策略。
- 不追求“替用户做决定”。

---

## 2. RALPLAN-DR 摘要
### Principles
1. **证据优先**：所有结论必须可回溯到来源、时间戳、抽取内容。
2. **安全优先**：只做决策支持，不越界到自动交易或隐性投资建议。
3. **范围收敛**：V1 只做有限标的/主题覆盖，先把链路做扎实。
4. **离线可验证**：关键流程可用离线样例与夹具验证，不依赖实时付费源。
5. **人工在环**：倾向分数只能辅助判断，最终决策必须由用户确认。

### Top 3 Decision Drivers
1. **合规/安全边界**：必须避免自动交易与不透明荐股风险。
2. **可落地性**：greenfield + 免费源 + 非实时，决定了 V1 需要收敛范围。
3. **可审计性**：若不能解释“为什么得出这个倾向”，产品不可用。

### Viable Options
#### 方案 A：研究摘要优先型（推荐）
**定义**：先把信息采集、证据归档、摘要、基础倾向输出做稳，再保守加入持仓视角分析。
- 优点：范围清晰；安全性高；更适合免费源与离线验证；最符合当前 deep-interview 目标。
- 缺点：个性化组合分析深度有限；早期“决策支持感”不如重分析方案强。

#### 方案 B：持仓分析优先型
**定义**：先围绕用户已持仓构建成本/仓位/风险/事件联动分析，再补外部信息聚合。
- 优点：对单个用户价值更直接；更容易体现“持仓相关建议”。
- 缺点：强依赖用户数据输入；外部证据底座不稳时，分析容易失真；早期泛化能力弱。

#### 方案 C：主题/板块雷达优先型
**定义**：先围绕行业/概念/主题做信息聚合与热度/事件跟踪，再映射到有限个股。
- 优点：适合 A 股题材驱动特点；对观察名单扩展友好。
- 缺点：主题噪声高；结论漂移风险大；需要更强去重与证据冲突处理。

### Recommended Plan
采用**方案 A，兼容少量持仓视角字段**：
- 主线先打通“有限股票池的信息证据链 + 摘要 + 倾向评分 + 审计输出”；
- 持仓分析只做 V1 基础能力，不做复杂组合优化；
- 主题/板块作为辅助上下文，不作为主判断引擎。

## 3. 团队 lane / 角色职责设计

### Lane 1：产品/需求负责人（Product / Requirements Owner）
**职责**
- 维护 `PROJECT.md` 的需求、范围、变更、决策、开放问题。
- 锁定 V1 边界：不自动交易、不全市场、不付费、不实时。
- 定义对外输出模板、确认点、里程碑与验收口径。
- 维护 `AGENTS.md` 后续治理要求：未来 agent 工作前后必须读/写 `PROJECT.md`。

**输入**
- deep-interview spec
- 用户补充偏好
- 各 lane 的能力/限制反馈

**输出**
- PRD / 需求基线
- 决策日志
- 范围变更记录
- 待确认项列表

**完成标准**
- V1 边界与非目标无歧义。
- 所有“需要用户确认”的事项被单独列出。

---

### Lane 2：宏观/市场信息源研究（Macro Market Source Researcher）
**职责**
- 梳理免费公开可用的市场级信息源：指数、宏观新闻、政策公告、交易所公告、上市公司公告入口等。
- 评估来源属性：可访问性、更新频率、结构化程度、许可证/使用限制、稳定性。
- 输出候选优先级与替代源。

**输入**
- 产品范围（覆盖哪些信息类型）
- 安全边界（免费、非实时）

**输出**
- 宏观/市场源清单
- 字段可得性矩阵
- 更新频率建议
- 风险/限制说明

**完成标准**
- 至少能支持：大盘/指数背景、政策/公告/新闻摘要上下文。
- 清楚标注哪些源只能作为参考，哪些可作为主证据。

---

### Lane 3：行业/主题/板块信息源研究（Sector / Theme Source Researcher）
**职责**
- 梳理行业、概念、板块、题材、公司相关新闻/公告等免费来源。
- 明确主题级信号如何映射到有限观察池个股。
- 识别高噪声源、重复源、营销型内容源。

**输入**
- 观察池/主题范围定义
- 宏观源研究结果

**输出**
- 板块/主题源地图
- 主题->个股映射规则建议
- 高噪声风险清单

**完成标准**
- 能为观察池个股提供行业/主题背景补充。
- 主题类来源被分级，不与主证据混同。

---

### Lane 4：数据工程/证据归档（Data Engineering）
**职责**
- 设计采集、解析、标准化、去重、快照留存、批次更新、审计链路。
- 设计统一证据记录模型：来源、抓取时间、原文片段、实体关联、批次号。
- 设计失败重试、源不可用回退、字段缺失标注。

**输入**
- 已确认源清单
- 产品定义的观察池/更新频率
- 分析 lane 需要的标准字段

**输出**
- 数据流设计
- 原始/标准化记录 schema
- 去重与冲突处理规则
- 批处理作业边界

**完成标准**
- 任一摘要/分析结果都能追溯到原始证据快照。
- 去重、缺失、冲突都有可解释规则。

---

### Lane 5：分析框架/评分设计（Analysis Model Designer）
**职责**
- 定义摘要结构：事实、事件、影响方向、证据强弱、反证、不确定性。
- 设计方向性倾向分数：区间、含义、阈值、解释文本。
- 区分“已持仓”和“未持仓/观察中”两类分析路径。
- 定义何时只能输出“信息不足/不形成倾向”。

**输入**
- 标准化证据数据
- 产品输出模板与安全约束

**输出**
- 分析 schema
- 评分规则草案
- 持仓/非持仓分析差异规则
- 反证与不确定性展示规范

**完成标准**
- 不出现命令式买卖语句。
- 每个分数都有证据、反证、置信度、不确定性解释。

---

### Lane 6：安全/合规审查（Safety / Compliance Reviewer）
**职责**
- 审查输出是否越界为投资建议或自动交易。
- 审查免责声明、人工确认、审计字段、证据透明度是否齐全。
- 审查“高不确定性时降级输出”的规则是否存在。

**输入**
- 产品输出模板
- 分析 schema 与示例输出
- 数据链路与审计字段定义

**输出**
- 安全检查清单
- 风险发现与整改要求
- 拒绝输出/降级输出条件

**完成标准**
- 所有高风险输出都有拦截或降级机制。
- 缺少证据/时间戳/免责声明的输出不能过审。

---

### Lane 7：测试/验收工程（Test / Verification Engineer）
**职责**
- 设计离线 fixtures、模拟源、回归样例。
- 设计端到端验收样例：从输入观察池/持仓到最终摘要/倾向报告。
- 覆盖失败场景：源不可用、证据冲突、字段缺失、噪声过高、分数被降级。

**输入**
- 全部 lane 的 schema、规则、输出模板

**输出**
- Test Spec
- Fixtures 设计清单
- 验收案例矩阵

**完成标准**
- 不依赖实时或付费数据也能完成主要验证。
- 关键风险场景都有测试样例。

## 4. V1 具体功能与需求

### 4.1 观察范围管理
**功能**
- 支持有限股票池输入：持仓、观察名单、选定主题/行业映射标的。
- 支持基础持仓信息输入：成本、仓位、建仓日期（至少预留）。

**要求**
- V1 必须拒绝“扫描全市场全部 A 股”的默认需求。
- 观察池变更需进入审计记录。

### 4.2 信息采集与证据归档
**功能**
- 批量采集/导入免费公开来源内容。
- 留存原始快照、抓取时间、来源标识、解析状态。

**要求**
- 至少支持文本类证据的统一归档。
- 同一事件多源重复需可合并或标记重复。

### 4.3 结构化摘要
**功能**
- 把原始信息转成结构化摘要：事件、主体、时间、可能影响、证据强弱。

**要求**
- 摘要必须保留来源链接/标识与时间戳。
- 若证据不足，摘要必须显式写“不确定/需补证”。

### 4.4 持仓相关分析
**功能**
- 对已持仓标的，结合成本/仓位/事件变化做风险-机会摘要。

**要求**
- 不做自动止盈止损指令。
- 只输出“值得关注的风险/机会点 + 倾向分数 + 人工确认提示”。

### 4.5 非持仓观察分析
**功能**
- 对观察名单标的给出基于证据的方向性倾向参考。

**要求**
- 不能输出“建议立即买入/卖出”。
- 分数必须附置信度、反证、数据缺口。

### 4.6 行业/主题上下文补充
**功能**
- 给个股结果补充板块/题材/行业层面的背景摘要。

**要求**
- 主题信号只能作为辅助上下文，不能单独主导结论。

### 4.7 审计与回溯
**功能**
- 记录每次分析的输入、用到的证据快照、规则版本、输出结果、时间。

**要求**
- 任一输出都可追溯其证据来源与分析版本。

### 4.8 输出模板
**最小输出结构**
- 标的/主题
- 分析时间
- 核心证据摘要
- 正向因素
- 负向因素/反证
- 数据缺口/不确定性
- 持仓视角说明（如适用）
- 倾向分数（非二元）
- 免责声明
- 人工确认提醒
- 审计 ID / 批次 ID

## 5. Lane 间输入 / 输出 / 交接

### 5.1 主交接链路
1. **Product** 定义范围、观察对象、确认点 -> 输出 PRD 基线。
2. **Macro Source** / **Sector Source** 基于 PRD 分头产出源清单与限制说明。
3. **Data Engineering** 根据已选/候选源设计统一证据 schema、更新批次、去重规则。
4. **Analysis Designer** 消费标准化证据 schema，定义摘要与评分框架。
5. **Safety Reviewer** 审查输出模板、评分语言、降级规则、免责声明。
6. **Test Engineer** 根据以上工件制定 fixtures、验收矩阵、失败场景。
7. **Product** 汇总待确认项，向用户做关键决策确认。

### 5.2 关键接口定义
#### Product -> Source Research
- 输入：V1 范围、优先信息类型、允许/禁止边界
- 输出：源研究任务说明书

#### Source Research -> Data Engineering
- 输入：来源清单、字段可得性、更新频率、限制说明
- 输出：可实现的数据接入候选矩阵

#### Data Engineering -> Analysis
- 输入：标准化记录 schema、证据链字段、实体映射规则
- 输出：分析可消费的数据契约

#### Analysis -> Safety
- 输入：分析输出 schema、分数定义、示例报告
- 输出：待审查高风险点清单

#### Safety -> Product / Analysis
- 输入：审查结论、必须整改项、拒绝条件
- 输出：输出边界修订清单

#### All lanes -> Test
- 输入：规则、schema、示例、失败条件
- 输出：可执行验收案例与回归清单

### 5.4 跨 lane 最小数据契约（新增）
#### evidence_record 最小字段
- `evidence_id`：证据唯一标识
- `source_id`：来源标识
- `source_type`：公告 / 新闻 / 宏观 / 行业 / 主题 / 其他
- `captured_at`：采集时间
- `event_time`：事件时间；未知时必须显式标 `unknown`
- `entity_scope`：市场 / 行业 / 主题 / 个股
- `entity_id`：标的代码或主题标识
- `content_hash`：去重键
- `provenance_level`：主证据 / 辅证据 / 噪声观察
- `license_note`：许可/使用备注（仅允许免费公开来源；不含新增付费/订阅源）
- `batch_id`：批次号
- `audit_id`：审计链标识

#### analysis_record 最小字段
- `analysis_id`
- `target_scope`：持仓 / 观察名单 / 主题补充
- `target_id`
- `used_evidence_ids`
- `summary`
- `positive_factors`
- `negative_factors`
- `counter_evidence`
- `uncertainty_notes`
- `score_state`
- `score_value`：无分数时允许为空
- `confidence_level`
- `disclaimer_block`
- `human_confirmation_required`：固定为 true
- `analysis_version`
- `generated_at`
- `audit_id`

#### handoff_bundle 最小字段
- `bundle_id`
- `producer_lane`
- `consumer_lane`
- `schema_version`
- `records_count`
- `validation_status`
- `known_gaps`
- `created_at`

### 5.5 评分资格/降级状态机（新增）
#### 状态定义
1. `summary_only`：仅做事实摘要，不进入倾向判断。
2. `no_score`：可做分析说明，但明确“不形成方向性分数”。
3. `low_confidence_score`：允许低置信倾向分数，必须强化反证与不确定性展示。
4. `normal_score`：允许正常倾向分数，但仍需免责声明、人工确认、审计记录。
5. `reject_output`：禁止生成可展示倾向输出，只能返回阻断原因与待补条件。

#### 触发原则
- 存在以下任一情况，至少降到 `no_score`：
  - 缺少可用时间戳
  - 仅有主题/板块噪声，无个股级或公司级证据
  - 证据冲突过大且无法解释主次
  - 证据数量不足或来源单一且低可信
- 满足以下条件，最多到 `low_confidence_score`：
  - 有个股级证据，但时效一般、反证明显、或数据缺口较大
- 出现以下任一情况，直接进入 `reject_output`：
  - 来源许可不合规
  - 审计字段缺失且无法补全
  - 免责声明/人工确认字段缺失
  - user_confirmation_gate 未通过但请求进入正式展示输出
- 只有在以下条件同时满足时，才可到 `normal_score`：
  - 至少存在一类个股/公司级有效证据
  - 证据时间有效且可回溯
  - 反证已展示
  - 来源许可合规
  - 安全审查字段完整

#### reject_output 的 lane handoff
- 责任 lane：Safety / Compliance Reviewer 拥有最终阻断权；Data Engineering 与 Analysis 负责补齐缺失字段或降级。
- 交接去向：
  - 许可问题 -> Source Research / Product
  - 审计缺失 -> Data Engineering
  - 免责声明/模板缺失 -> Analysis / Product
  - 用户确认未过 -> Product 待确认队列

### 5.6 来源治理包（新增）
每个来源研究 lane 交付的不只是“候选源清单”，还必须包含：
- allowlist：允许纳入 V1 候选的免费公开来源
- denylist：明确禁用或高风险来源
- forbidden_methods：禁止抓取/使用方式
- license_note：许可、署名、使用限制
- freshness_note：该来源适合的更新节奏
- provenance_level_default：默认证据级别
- user_confirmation_gate：是否必须在最终接入前用户确认

### 5.7 主题/板块 -> 个股映射硬约束（新增）
- 主题/板块信号只能作为上下文增强层。
- **没有个股级/公司级证据时，主题信号不得单独驱动任何带分数输出。**
- 若只有主题证据，最高只能到 `no_score`，且必须标注“主题映射推断”。
- 主题映射进入最终摘要时，必须附映射依据与不确定性说明。

### 5.8 V1 交付表面与 freshness 口径（新增）
#### 交付表面
V1 规划默认交付形态为：
- 批次运行输入：有限观察池 / 持仓 / 主题列表
- 批次运行输出：
  - 单标的 Markdown 报告
  - 机器可读 JSON 摘要包
  - 审计记录索引
- 最终使用方式：用户阅读报告后人工确认，不触发自动动作

#### freshness 口径
- 默认以 **T+0 盘后** 或 **T+1 日级批次** 为主。
- 若做盘中批次，也只视为低频批次更新，不承诺实时。
- 可测试阈值（V1 默认）：
  - `company_announcement`：`effective <= 3 个自然日`；`stale > 7 个自然日`
  - `market_news`：`effective <= 2 个自然日`；`stale > 5 个自然日`
  - `macro_policy`：`effective <= 5 个自然日`；`stale > 14 个自然日`
  - `sector_theme`：`effective <= 1 个自然日`；`stale > 3 个自然日`
  - `unknown_time`：不得高于 `no_score`
- 规则：
  - `effective` 可参与正常状态判断
  - `aging`（介于 effective 与 stale 之间）最高到 `low_confidence_score`
  - `stale` 或 `unknown_time` 不得支持 `normal_score`

## 6. 验收标准（Acceptance Criteria）

### 6.1 规划层验收
- 已形成覆盖 7 个 lane 的职责、输入、输出、交接定义。
- 已形成 V1 功能清单，并逐项标注边界约束。
- 已形成最小输出模板与最小审计字段集合。
- 已形成“必须用户确认”的决策清单。

### 6.2 产品能力验收（V1 目标）
- 系统只面向有限观察池/持仓/主题范围。
- 任一输出都包含证据来源与时间戳。
- 任一方向性倾向都包含：分数、置信度、不确定性、反证、免责声明、人工确认、审计标识。
- 缺证据或冲突过大时，系统可降级为“不形成倾向”。
- 不存在自动下单、自动交易、券商执行接口。
- 不依赖付费或高频实时源完成 V1 核心闭环。

### 6.3 测试/验证验收
- 有离线 fixtures 覆盖正常、缺失、冲突、重复、噪声等场景。
- 至少有 1 条持仓分析样例、1 条观察名单样例、1 条主题补充样例。
- 至少有 1 条“拒绝输出/降级输出”样例。

## 7. 主要风险与应对

### 风险 1：免费源质量与稳定性不足
- 影响：摘要偏差、证据断裂、更新不稳定。
- 应对：多源分级；主证据/辅证据区分；源失效回退方案；缺失显式暴露。

### 风险 2：主题/题材信号噪声过高
- 影响：错误放大市场情绪，导致倾向失真。
- 应对：主题源降权；要求个股级证据共振后才提高分数；保留反证位。

### 风险 3：输出越界成投资建议
- 影响：产品合规与安全风险高。
- 应对：强制免责声明、非命令式语言、人工确认、无自动交易接口、审查 gate。

### 风险 4：证据与结论无法回溯
- 影响：用户无法信任系统，也无法调试错误。
- 应对：统一审计 ID、证据快照、规则版本、时间戳、批次 ID。

### 风险 5：greenfield 范围膨胀
- 影响：V1 迟迟不能落地。
- 应对：只做有限池；先做摘要闭环；高级组合分析延后。

### 风险 6：持仓数据敏感性
- 影响：用户顾虑输入隐私，影响使用。
- 应对：V1 先按最小字段设计；持仓输入作为可选项；后续再细化隐私方案。

## 9. ADR（Architecture Decision Record）
### Decision
采用“**研究摘要优先 + 有限持仓视角补充**”的 V1 方案，围绕有限股票池建立证据链、摘要、方向性倾向与审计闭环。

### Drivers
- 必须遵守不自动交易、非实时、免费源、非全市场边界。
- 需要在 greenfield 条件下优先构建最可验证的闭环。
- 需要让任何倾向输出都可解释、可追溯、可降级。

### Alternatives Considered
- **持仓分析优先**：用户价值直接，但过度依赖用户数据，证据底座不足时风险高。
- **主题雷达优先**：贴近 A 股题材特征，但噪声大、易漂移，不适合作为 V1 主路径。

### Why Chosen
该方案最符合已确认用户目标，也最容易在免费、非实时、有限范围内做出可信演示与后续扩展底座。

### Consequences
- 早期个性化深度有限，但系统可信度更高。
- 数据工程与审计设计需要前置，不宜只做“文本摘要表面功能”。
- 需要明确源研究 lane 与安全 lane 的 gate 作用。

### Follow-ups
- 后续可在用户确认后扩展：持仓风险分析深度、主题雷达能力、更多结构化财务指标、更多交互界面。
