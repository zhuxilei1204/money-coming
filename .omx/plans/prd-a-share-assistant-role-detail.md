# PRD: A股交易辅助帮手 V1 角色与功能细化

## 1. 背景与目标
本轮 `$ralplan` 的目标是进一步明确每个 team 角色关注点、职责、输入输出、交接关系，以及 V1 具体功能和要求；不直接创建生产功能、不接入数据源、不修改交易相关系统。项目 V1 是 A 股交易辅助工具：优先做信息聚合摘要，可在严格护栏下输出评分式买卖倾向，但不自动交易。

## 2. RALPLAN-DR 摘要
### Principles
1. 证据优先：任何摘要、评分、倾向必须能追溯到来源、时间和处理版本。
2. 辅助而非代替决策：不下单，不给命令式建议，保留人工确认。
3. 小范围可验证：先覆盖持仓/自选/指定板块，不做全市场扩张。
4. 免费/授权优先：V1 不接入付费源；候选源必须记录限制。
5. 项目记忆优先：规划需定义 `PROJECT.md` 维护规则，实际创建/更新放入后续执行。

### Decision Drivers
1. 安全合规边界：评分式倾向必须绑定证据、不确定性、免责声明、审计记录。
2. 数据源可持续性：免费/授权源是否稳定，是 V1 能否成立的核心风险。
3. Team 可并行交付：角色边界必须减少依赖冲突，便于后续 `$team` 执行。

### Viable Options
- **Option A: 角色/功能/测试细化先行（推荐）**
  - Pros: 对齐当前“进一步讨论”目标；不越过用户确认门；适合绿地仓库。
  - Cons: 不会立即验证真实数据源可用性。
- **Option B: 源可行性验证先行**
  - Pros: 更快暴露免费源是否可用。
  - Cons: 可能在角色职责和安全模板未定前过早进入实现倾向。
- **Option C: 快速原型先行**
  - Pros: 很快看到 demo。
  - Cons: 违背“技术栈/模板/源接入需确认”的边界，返工风险高。

Decision: 采用 Option A，并在规划末尾设置非实现型 feasibility gate：下一阶段先确认数据源候选与模板，再决定实现。

## 3. V1 具体功能清单
### 3.1 目标对象
- 持仓股票：用户已买入标的，可结合成本、仓位、持有周期、风险暴露。
- 自选股票：用户关注但未持仓标的，只做研究/观察倾向。
- 指定板块/主题：用于宏观和板块方向摘要，不扩展到全 A 覆盖。

### 3.2 信息聚合功能
- 输入：股票代码/名称、板块/主题、日期范围、用户持仓上下文（可选）。
- 输出：聚合摘要列表，每条包含最小字段集：标题、来源、发布时间、抓取/记录时间、关联标的/板块、事件类型、摘要、证据链接或引用、数据质量标记。
- 约束：只研究免费或用户已有权限源；不做实时高频；不接入付费源。

### 3.3 单次分析功能
- 输入：标准化摘要记录、行情/公告/研报类信息、持仓上下文（可选）。
- 输出：事实摘要、风险/机会、反方因素、不确定性、置信度、适用前提、失效条件、评分式方向倾向、免责声明、人工确认提示。
- 评分：允许方向评分，但具体区间、权重、阈值必须后续确认；禁止命令式/二元化“买/卖”话术。

### 3.4 审计记录功能
最小字段：输入数据快照 ID、来源 URL/引用、时间戳、分析模板版本、评分规则版本、输出结果、人工确认状态、运行人/agent、错误或缺口。

### 3.5 人工确认触发点
以下事项未确认前不得实现或接入：技术栈与目录结构、具体数据源接入、评分公式与模板、首版交互形态、任何付费源或更高频更新。

## 4. 非功能要求
- 可追溯：每个结论都能回到输入源和模板版本。
- 可测试：默认使用离线 fixture/mocks，不依赖付费或实时源。
- 可替换：数据源适配器与分析模板应解耦。
- 可审查：安全合规 checklist 必须在实现前定义。
- 可维护：后续执行阶段应创建 `PROJECT.md` 并更新 `AGENTS.md` 规则；本轮只定义结构和维护要求。

## 5. 角色与泳道职责
### 5.1 产品 / 需求负责人
关注点：范围控制、需求澄清、项目记忆、变更治理。
输入：deep-interview spec、本 PRD、用户新增要求、各角色交付物。
任务：定义 `PROJECT.md` 结构与维护规则；定义后续 `AGENTS.md` 约束；维护 V1/非 V1、决策日志、风险、待确认项。
输出：项目事实源结构、变更记录规范、验收清单、待确认问题表。
交接：向所有角色提供当前范围与约束；向安全/测试提供验收依据。

### 5.2 宏观市场数据源研究员
关注点：大盘、指数、政策、宏观新闻、市场情绪、流动性、行业整体信号。
输入：V1 范围、免费/授权约束、宏观信息需求、候选源评价模板。
任务：列出免费/授权候选源；记录字段、频率、访问方式、限制、稳定性、合规注意点；区分机器抓取与人工导入。
输出：宏观源候选报告结构，含 source matrix 与风险说明。
交接：给数据工程做可接入性评估；给分析模型设计者提供宏观特征候选。

### 5.3 板块 / 方向数据源研究员
关注点：行业、概念、主题、板块新闻、板块成分、板块热度与事件。
输入：关注板块/主题范围、免费/授权约束、板块分类需求、候选源评价模板。
任务：梳理板块/主题相关免费源；记录分类差异、覆盖缺口、字段、频率、可追溯链接。
输出：板块/主题源候选报告结构，含 coverage map 与不可自动化项。
交接：给数据工程和分析模型设计者；标明不可稳定自动化的来源。

### 5.4 数据工程执行者
关注点：可测试、可追溯、可替换的数据管道。
输入：已确认的数据源、source matrix、字段需求、审计字段要求、fixture 规范。
任务：后续实现抓取/导入、解析、去重、标准化、存储、审计字段；本轮只定义接口和数据模型候选。
输出：数据模型草案、ingestion interface 草案、审计日志结构草案。
交接：给分析模型提供标准化 records；给测试工程师提供 fixture 生成方式。

### 5.5 分析模型设计者
关注点：摘要结构、证据链、风险/机会标签、评分体系、持仓/未持仓差异。
输入：标准化记录草案、宏观/板块特征候选、用户持仓上下文、强制安全护栏。
任务：提出分析 schema、评分维度、权重草案、置信度与反方因素表达；设计样例输出。
输出：分析 schema 草案、评分模板草案、持仓/未持仓样例。
交接：需用户确认后才能实现；交给安全角色审查措辞和护栏。

### 5.6 安全 / 合规审查者
关注点：不自动交易、不越界投顾、可追溯、免责声明、人工确认。
输入：分析模板草案、评分输出样例、审计日志字段、非目标约束。
任务：检查输出不含命令式买卖；确认每个评分有证据、不确定性、免责声明、审计记录、适用前提和失效条件。
输出：安全 checklist、禁止话术列表、必备字段清单。
交接：向测试角色提供必须验证的安全断言；向分析设计者反馈模板修正。

### 5.7 测试 / 验收工程师
关注点：离线可测、回归稳定、验收可执行。
输入：PRD、test spec、fixture 规范、分析样例、安全 checklist。
任务：创建 fixture 设计、mock 源设计、回归测试清单；验证非目标不被突破。
输出：测试计划、fixture schema、验收脚本需求。
交接：向团队汇总验证证据；阻止无证据结论进入完成状态。

## 6. 待用户确认项
1. 技术栈与目录结构。
2. 实际接入的数据源和优先级。
3. 评分区间、权重、阈值和展示模板。
4. 持仓数据输入格式：CSV/JSON、表格、手动录入、或后续 UI。
5. 首版交互形态：CLI、Markdown 报告、Notebook、Web UI 或定时报告。

## 7. 下一阶段建议
1. 先由用户确认本 PRD 中角色职责、V1 功能清单和确认门。
2. 后续执行阶段再创建 `PROJECT.md`、更新 `AGENTS.md`，并启动两条数据源研究线。
3. 若进入并行执行，使用 `$team`；若需要 durable 目标跟踪，使用 `$ultragoal` + `$team`。

## 8. ADR
Decision: 本轮只细化角色、功能、非功能要求、交接关系和验收门，不直接创建治理文件或实现数据接入。
Drivers: 用户要求继续讨论细节；技术栈/源/模板需确认；金融辅助系统需要安全边界。
Alternatives: 源验证先行、快速原型先行。前者适合下一阶段，后者当前风险过高。
Consequences: 规划更稳但实现延后；下一阶段必须尽快做数据源可行性验证。
Follow-ups: Architect/Critic 通过后，向用户提交可讨论版本；确认后再 `$team` 或 `$ultragoal`。
